Technical deep-dive

Every question answered precisely

Why both surprise AND reward?

They answer completely different questions and operate at different times. Removing either one breaks the system in a distinct way.

Surprise metric

Runs during the forward pass, token by token. Answers: "is this token worth writing into memory at all?"

Gradient norm of associative loss. High = model doesn't already know this association. Low = model already has it. Gates the write strength.

Reward-conditioned update

Runs after task completion, once per task. Answers: "did the action that came from this memory state actually work?"

TD error between predicted reward and observed reward. Updates which (situation→action) associations are reinforced or suppressed.

What breaks if you remove one
Without surprise (only reward): Memory writes every single token with equal weight. Within one 500-token problem, all 500 tokens update the weights, most of them carrying zero signal about what mattered. Memory saturates with noise. The weights stop being useful — they're averaging over irrelevant tokens like "def", "return", "the". Result: reward signal is correct but buried in noise.
Without reward (only surprise): Memory writes what's surprising — novel syntax, unusual patterns, unexpected phrases. But "surprising" is not the same as "what led to success." The model might strongly memorise an unusual bug it introduced and then keep repeating it, because it's consistently surprising but never gets corrected. Result: memory learns interesting things, not useful things.
How they interact — the two-phase write
// PHASE 1 — during forward pass (surprise-gated associative write):
// For each token t in the problem+solution sequence:
surprise_t = ‖∇_θ ‖M_θ(k_t) − v_t‖²‖
// Only write if surprise exceeds threshold
if surprise_t > τ:
  θ ← θ − η_assoc · surprise_t · ∇_θ loss

// PHASE 2 — after reward arrives (reward-conditioned update):
td_error = r_actual − M_θ(encode(situation, action))
θ ← θ + η_reward · td_error · ∇_θ M_θ(encode(situation, action))

// Phase 1 decides WHAT gets stored (novelty filter)
// Phase 2 decides HOW STRONGLY to weight it (outcome filter)
The intuition: Surprise is the filter at the door — only let in things that are new. Reward is the reinforcement signal inside — of the new things that came in, strengthen what worked and weaken what failed. You need both because new-but-wrong and old-but-right are both useless on their own.
What exactly are state, action, and reward here?

This is the hardest question to answer correctly. You're right that there's no ready-made tuple. The model doesn't know "state" or "action" — it only produces tokens. Here's how we construct them.

The fundamental problem
In classical RL, state and action are explicit: the game board is the state, moving left is the action. In an LLM generating code, neither is explicit. The "action" is an entire sequence of tokens (the generated solution). The "state" is whatever the model was "thinking" when it decided to generate that sequence. We need to construct both from the model's own hidden representations.
State — the situation encoding
State = average hidden representation while reading the problem

When the model processes the problem description (before generating anything), its hidden states at the memory layer form a representation of "the situation." We take the mean of those hidden states as the state vector.

// Process problem description, extract hidden states at memory layer:
problem_tokens = tokenize("Write a function that reverses a string...")
hidden_states = model.get_hidden_states(problem_tokens) # shape: [seq_len, dim]

state_vec = hidden_states.mean(dim=0) # shape: [dim]
// This is the model's compressed "understanding" of the problem situation.
// Similar problems will have similar state_vec values — cosine similarity > 0.8
// Very different problems will have distant state_vecs
The key property you need: two problems that require similar solutions should have similar state vectors. This is what allows the memory to generalise — "I failed at a string reversal problem before, and this looks like another string reversal problem." That generalisation happens automatically because the embedding space clusters similar problems together.
Action — the solution encoding
Action = average hidden representation while generating the solution

After the model generates its code output, we pass the generated tokens back through the model and extract the hidden states. This gives us a vector representation of "what the model did."

// After generation, encode the solution:
solution_tokens = tokenize(generated_code)
solution_hidden = model.get_hidden_states(solution_tokens) # [seq_len, dim]

action_vec = solution_hidden.mean(dim=0) # shape: [dim]

// Combine state + action into one encoding for the memory:
situation_action = torch.cat([state_vec, action_vec]) # shape: [2*dim]
// Or use a learned projection:
situation_action = W_sa · torch.cat([state_vec, action_vec]) # shape: [dim]
Reward — the outcome signal
Reward = scalar from the execution sandbox

This is the clean part. Run the code. Count passed tests. Normalize to [0, 1].

reward = tests_passed / total_tests

// Examples:
// All 3 tests pass → reward = 1.0 (perfect)
// 2 of 3 tests pass → reward = 0.67 (partial credit — important!)
// Syntax error → reward = 0.0 (complete failure)
// Timeout → reward = −0.1 (worse than wrong — penalise loops)
The complete TD update — with real vectors
// Memory predicts reward for this (state, action) pair:
predicted_r = M_θ(situation_action).mean() # scalar

// Observe actual reward from sandbox:
actual_r = tests_passed / total_tests

// TD error — how wrong was the memory's prediction?
td_error = actual_r − predicted_r

// Update memory weights to reduce this error next time:
loss = (td_error)² = (actual_r − M_θ(situation_action))²
θ ← θ − η · ∇_θ loss

// Next time same situation_action appears → M_θ predicts closer to actual_r
// For unseen-but-similar situations → M_θ generalises via weight geometry
The crucial insight: The memory is learning the function: f(situation, action) → expected_reward. Once trained, before generating, the model can query: "for this situation, which action encoding gives me the highest expected reward?" That's the RL loop. During generation (beam search or sampling), action candidates with higher predicted reward get higher weight. The memory has become a value function — embedded in the weights of M_θ.
Building and testing the memory module in isolation

"In isolation" means: no LLM. No transformer. No tokenizer. Just the NeuralMemoryModule class, synthetic data, and assertions. You're verifying the mechanism works before attaching it to anything expensive.

What you're actually testing — 4 properties
1. Associative storage
Write (k₁→v₁), then query k₁. Should recover v₁ with low error. Like a hashtable but in weight space.
2. Surprise behaviour
Surprise should be high on first write of a new key. Near-zero on repeated writes. This verifies the gradient norm signal works.
3. Reward learning
After reward updates for (s,a)→r pairs, M_θ(s,a) should approach r. Verifies TD update propagates into weights.
4. Generalisation
After learning (s₁→r=1.0) and (s₂→r=0.0), a new s₃ similar to s₁ should predict closer to 1.0. This is the key property.
5. Decay
Old memories should fade if not reinforced. After 100 unrelated writes, the original k₁ retrieval should show higher error.
6. Interference limit
Writing 500 different (k,v) pairs shouldn't cause catastrophic forgetting of the first 10. Measures memory capacity.
Concrete isolation test suite
import torch
import torch.nn as nn

def test_associative_storage(mem, dim=64):
  """Test 1: memory can store and retrieve a key-value pair."""
  k = torch.randn(dim) # a fixed key
  v = torch.randn(dim) # the value to associate

  # Force memory to overwrite W_K/W_V to point at this pair
  # by writing it 50 times (simulate repetition)
  for _ in range(50):
    mem.write_kv(k, v) # direct k,v write, no projection

  retrieved = mem.read(k)
  error = ((retrieved - v)**2).mean().item()
  assert error < 0.1, f"Storage failed: error={error:.4f}"
  print(f"✓ Associative storage: error={error:.4f}")

def test_surprise_behaviour(mem, dim=64):
  """Test 2: surprise high on first write, low on repeated writes."""
  k = torch.randn(dim)
  v = torch.randn(dim)

  surprises = []
  for i in range(20):
    s = mem.write_kv(k, v)
    surprises.append(s)

  assert surprises[0] > surprises[-1] * 3, "Surprise should decay with repetition"
  print(f"✓ Surprise decay: {surprises[0]:.3f} → {surprises[-1]:.3f}")

def test_reward_learning(mem, dim=64):
  """Test 3: memory learns to predict reward from (state, action) pairs."""
  # Create 3 fixed (state, action) → reward pairs
  pairs = [
    (torch.randn(dim), 1.0), # this approach works
    (torch.randn(dim), 0.0), # this approach fails
    (torch.randn(dim), 0.5), # this approach partially works
  ]

  # Train memory on these pairs for 100 steps
  for _ in range(100):
    for enc, reward in pairs:
      predicted = mem.layers(enc).mean().item()
      mem.reward_update(enc, predicted, reward)

  # Verify predictions are close to actual rewards
  for enc, expected_r in pairs:
    predicted = mem.layers(enc).mean().item()
    assert abs(predicted - expected_r) < 0.15, \
      f"Reward prediction failed: got {predicted:.3f}, expected {expected_r}"
  print("✓ Reward learning: predictions within 0.15 of actual")

def test_generalisation(mem, dim=64):
  """Test 4: similar situations → similar predicted reward."""
  base = torch.randn(dim)
  similar = base + 0.1 * torch.randn(dim) # small perturbation
  different = torch.randn(dim) # unrelated vector

  # Teach: base → reward 1.0
  for _ in range(50):
    mem.reward_update(base, 0.0, 1.0)

  r_base = mem.layers(base).mean().item()
  r_similar = mem.layers(similar).mean().item()
  r_diff = mem.layers(different).mean().item()

  # Similar should predict closer to 1.0 than different
  assert abs(r_similar - 1.0) < abs(r_diff - 1.0), "Generalisation failed"
  print(f"✓ Generalisation: base={r_base:.2f}, similar={r_similar:.2f}, diff={r_diff:.2f}")

# Run all tests
mem = NeuralMemoryModule(dim=64)
test_associative_storage(mem)
test_surprise_behaviour(mem)
test_reward_learning(mem)
test_generalisation(mem)
All 4 tests passing = the memory module works as designed. Then and only then, attach it to the LLM. This saves you from debugging a broken memory module inside a 3B model where the error signal is invisible.
Problem difficulty calibration — the real problem

You're right to be skeptical. This is a known failure mode called the floor-ceiling problem in evaluation design, and it completely invalidates results if ignored.

Why easy and hard problems are both useless
Too easy (both models pass):
Session 1: Memory model 90%, Baseline 90%
Session 5: Memory model 91%, Baseline 90%

You can't see improvement. The problems were already solved. You can't measure learning where there's nothing to learn.
Too hard (neither model passes):
Session 1: Memory model 5%, Baseline 5%
Session 5: Memory model 6%, Baseline 5%

Reward is always ~0. The TD error is near-zero. The memory layer has no signal to learn from. Nothing improves.
What you need: problems the model solves 30–60% of the time in Session 1. These are problems where it sometimes gets the right approach and sometimes doesn't — meaning the reward signal is sometimes 1.0, sometimes 0.0, giving the memory real gradient to learn from. By session 5, you want to see 50–75% pass rate on these. That delta is your result.
How to calibrate — practical protocol
1
Empirical difficulty scan first

Before running any memory experiments, run the base model (no memory) on 60 HumanEval problems across 3 attempts each. Record the pass rate per problem. This gives you the model's inherent difficulty distribution for your chosen model.

2
Select the 30–60% band

Keep only problems where the base model passed 1 or 2 out of 3 attempts. Discard problems it always passed (too easy) and problems it never passed (too hard). This is your calibrated eval set. For Gemma 2B, this will be roughly HumanEval problems 20–50 by difficulty.

3
Use partial credit rewards

Binary pass/fail (0 or 1) is too coarse. Add partial credit: if a function handles 2 of 3 edge cases, reward = 0.67. This gives the memory richer gradient signal and reduces the variance of the reward estimate. The sandbox already supports this with tests_passed / total_tests.

4
Use problem clusters, not random selection

Pick 5 clusters of 4 structurally similar problems: string manipulation × 4, array operations × 4, recursion × 4, dictionary operations × 4, math/modular arithmetic × 4. This lets you measure whether the memory generalises within a cluster — learns "string reversal approach" not just "this specific problem."

The metric that actually matters
// Not just overall pass rate — measure improvement on FAILED problems

F = {problems that failed in Session 1}

improvement_rate = |{p ∈ F : passes in Session N}| / |F|

// This is the signal. Base model: improvement_rate ≈ 0 (no memory = no change)
// Memory model: improvement_rate should increase with each session
// Target: improvement_rate > 0.3 by session 5
Every failure mode — and how to detect + prevent each one

In rough order of likelihood. Most of these will happen. Knowing them in advance means you debug in minutes, not days.

① Gate collapse — model ignores memory entirely

What happens: The gate g_t = sigmoid(W_g · x) learns to output ≈ 0 for everything, using only the attention output. Memory layer is dead — it has no effect on output.

How to detect: Log g_t values during inference. If median g_t < 0.05 after 1000 steps, gate has collapsed.

How to fix: Add a gate regularisation loss: L_gate = (mean(g_t) − 0.3)² — penalise the gate for being too close to 0 or 1. Forces it to actually blend memory and attention.

② Memory saturation — all weights collapse to near-zero

What happens: The forget gate α_t ≈ 1 everywhere, decay term (1−α_t) ≈ 0, all weights drive to zero. Memory MLP outputs near-zero vectors for any query. Associative storage fails.

How to detect: Log ‖θ‖ (weight norm) each step. If it monotonically decreases toward 0, saturation is happening.

How to fix: Clip α_t to [0.0, 0.9] — never allow full decay. Add weight norm monitoring as a training assertion.

③ Gradient explosion in the write loop

What happens: Computing ∇_θ for every token causes gradients to accumulate over a 500-token sequence. By token 500, gradient norms are in the thousands. Memory weights blow up, outputs become NaN.

How to detect: NaN in outputs within 10 steps of starting the memory write. Or gradient norms > 100.

How to fix: Clip gradients before applying: grad.clamp_(−1.0, 1.0). Use a lower learning rate η = 0.001 not 0.01 for the memory write. Also: only write on tokens with surprise > threshold, which already caps the number of writes per sequence.

④ State encoding collapse — all problems look the same

What happens: The mean of hidden states over a 200-token problem description gives a nearly identical vector for all problems because the embeddings are dominated by shared tokens ("Write a function that...", "return", "def"). State generalises too aggressively — memory thinks all problems are the same situation.

How to detect: Compute cosine similarity between state_vecs of 20 different problems. If average similarity > 0.95, they're collapsing.

How to fix: Use the last hidden state rather than mean. Or use only the hidden states of the "content words" (skip stopwords at tokenisation). Or use a learned projection W_state that's trained to maximally separate problem types.

⑤ Reward sparsity — almost always 0, rarely 1

What happens: If your calibration is off and the model almost always fails, reward is almost always 0. TD error is near-zero. Memory weights barely update. No learning happens and you think the architecture is broken, but it's the problem selection.

How to detect: Mean reward per session < 0.1.

How to fix: This is why problem calibration (previous tab) is done first. If you see this in practice, switch to easier problems immediately. Don't debug the architecture when the problem is the data.

⑥ Session loading interference — memory hurts more than it helps

What happens: Memory from session 1 actively interferes with session 2 because the weight updates from wrong answers in session 1 are pointing in the wrong direction for different problems in session 2. Pass rate goes DOWN with memory.

How to detect: Session 2 pass rate < session 1 on the memory model.

How to fix: Only save memory updates from successful episodes (reward > 0.7). Failed attempts write noise. Alternatively, scale the TD update by the reward magnitude — zero reward → zero write. High reward → strong write.

⑦ Base model quality degradation from memory grafting

What happens: Inserting the MemoryAugmentedDecoderLayer changes the residual stream of the transformer. Even with g_t gated toward attention, the wrapper itself introduces floating point differences that compound across 32 layers. The model outputs gibberish even before any memory writes.

How to detect: Immediately after grafting (before any writes), run perplexity on a validation set and compare to pre-graft perplexity. Should be within 1%.

How to fix: Initialise gate bias to a large negative value (−5.0), which initialises g_t ≈ sigmoid(−5) ≈ 0.007 — memory layer is nearly invisible at init. Let it learn to activate gradually.

Multi-user memory and RL memory policy — explained properly
Multi-user memory — the problem
What it means and why it's hard

Right now the prototype assumes one user — you. One memory state θ that persists across your sessions. This is manageable. But as soon as you want multiple users, you have choices about where the memory lives and how it updates — and each choice has tradeoffs.

Option A: Per-user θ files
Each user gets their own memory_alice.pt, memory_bob.pt. Simple. Clean. Scales horizontally. The problem: the base model is shared but memory is private. If Alice and Bob both learn that "recursion works for tree problems", they each learn it separately. No knowledge sharing.
Option B: Shared θ + user embedding
One shared memory layer, but each query/write is conditioned on a user_id embedding. memory.read([q_t ‖ user_emb]). Allows knowledge sharing while maintaining personal context. Much harder to implement — user embeddings need to be learned.
Option C: Hierarchical θ
Two memory layers: a shared "world knowledge" layer (updated by all users) and a personal "user model" layer (updated by each user separately). Research frontier — nobody has built this cleanly.
For the prototype: Option A. One user. Full stop. Multi-user memory is a future research problem, not a prototype concern. The README puts it in future work because it's a real system design challenge — not because it's needed to prove the core idea.
RL training on memory policy — what it actually means
The difference between a fixed rule and a learned policy

In the prototype, the memory update rule is hand-coded:

// Prototype rule (hand-coded):
if surprise > threshold AND reward > 0.5:
  write strongly
elif reward < 0.1:
  suppress / decay
else:
  write weakly

This works. But "threshold", "0.5", "0.1" are all guesses. Different users, different domains, different task types all have different optimal thresholds. The hand-coded rule is a one-size-fits-all approximation.

RL training on memory policy means: instead of this hand-coded rule, train a small neural network (the policy π) to decide what to do. It takes as input (surprise_t, reward_t, age_of_memory, task_type, user_context) and outputs an action from {WRITE_STRONG, WRITE_WEAK, UPDATE, SUPPRESS, NOOP}.

// Learned policy (future work):
a_t = π_RL(surprise_t, reward_t, memory_age, task_embedding, ...)
a_t ∈ {WRITE_STRONG, WRITE_WEAK, UPDATE, SUPPRESS, NOOP}

// π_RL is trained with GRPO on a dataset of (memory_state, action, outcome)
// The reward for π is: did this memory operation improve future task performance?
// Memory-R1 (2025) proved this works for external memory.
// Applying it to internal weight-space memory = Paper 3.
For the prototype, this is not needed. The hand-coded rule is good enough to prove the core loop works. RL memory policy is what you do after the prototype works — when you want to optimise the management of what's already a functional memory system.
